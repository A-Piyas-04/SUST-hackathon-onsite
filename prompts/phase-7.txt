## Hard Guardrails (Never Violate)

1. Decision-support only. No real financial operations or punitive automation.
2. Never implement transfer/refill/recovery/reversal/freeze/block/accusation/fraud verdict actions.
3. Shared physical cash and provider e-money (bKash, Nagad, Rocket) remain separate — never blended in metrics or reports.
4. Provider boundaries are strict: Provider A users cannot read Provider B confidential records.
5. Validation evidence must be honest: state sample sizes, seeds, configuration, engine versions, and limitations explicitly.
6. Held-out evaluation data must be frozen before measurement — no tuning on the reported split.
7. Use safe wording everywhere: "unusual", "requires review", uncertainty-aware language — never definitive fraud claims.
8. Synthetic/demo-safe identities and data only; no real credentials, phone numbers, or account data.
9. Metrics and scans must reference the signed-off release-candidate identifier (commit hash, engine version, contract version).

## Implementation Plan (execute in this order)

### Step 1) Held-out evaluation harness

Build a validation service that evaluates the frozen MVP against held-out synthetic data.

1. **Freeze held-out split**
   - Use scenarios/runs with `validation_split = held_out` from `reference_seed.sql` (do not use `demo` or `tuning` splits for reported metrics).
   - Persist a `validation_run` row with `dataset_split = held_out`, `engine_version`, `configuration` (seed, thresholds, commit hash), and `status`.

2. **Ground truth labels**
   - Populate `ground_truth_labels` from known scenario expectations (shortage windows, anomaly clusters, normal negatives, data-quality incidents).
   - Link labels to `simulation_run_id`, `outlet_id`, and optional `provider_id`.
   - Label types: `shortage`, `anomaly`, `normal`, `data_quality_incident`.

3. **Analytics metrics (at least two from this category)**
   - `anomaly_precision` — true positives / (true positives + false positives) against injected held-out anomaly labels.
   - `anomaly_recall` — true positives / (true positives + false negatives).
   - `anomaly_false_positive_rate` — false positives / total negatives (or equivalent honest definition documented in `method`).
   - `forecast_mae_minutes` or `shortage_lead_time_minutes` — forecast error or shortage-detection lead time against held-out shortage labels.
   - Each metric row must include positive `sample_size`, explicit `method`, and honest `limitations` (e.g., synthetic data only, small sample, single rule).

4. **Evaluation runner**
   - Provide a documented command (e.g., `make validate` or `python -m app.cli validate`) that:
     - resets/runs held-out scenarios deterministically;
     - executes quality + liquidity + anomaly analytics;
     - compares outputs to ground truth;
     - persists `metric_results` and marks the validation run `completed` (or `failed` with reason).
   - Re-running with the same seed/config must produce the same metric values.

### Step 2) Performance and reliability metrics (at least one from this category)

Measure operational correctness on the frozen build:

1. **API latency**
   - `api_avg_ms` and/or `api_p95_ms` at documented request volume (state endpoint set and concurrency in `details`).
   - Use a repeatable load script or pytest benchmark against representative endpoints (dashboard, projections, alerts list).

2. **Reliability/completeness counters** (persist or expose via `/metrics`)
   - `data_quality_incident_rate` — share of feeds/windows classified stale/missing/conflicting vs total evaluated.
   - `alert_explanation_coverage` — fraction of published high-impact alerts with complete EN explanation sections (situation, evidence, uncertainty, next step).
   - `case_transition_completeness` — legal transitions exercised in Scenario D produce expected audit + notification side effects.
   - `provider_denial_correctness` — cross-provider access attempts denied consistently (count or pass/fail in `details`).
   - `idempotency_correctness` and `concurrency_correctness` — duplicate/stale mutations fail safely without duplicate history.

Ensure the **total persisted/reported numeric metrics ≥ 3** across analytics + performance/reliability categories, each with method, sample size, and limitations.

### Step 3) Observability endpoints

Replace stubs with persistence-backed implementations.

#### `GET /health` (verify existing)
- Liveness and database readiness; no confidential details.
- Confirm it still passes after integration.

#### `GET /metrics` (implement)

- Query parameters: optional `validation_run_id`, `dataset_split`, `status`.
- Response shape: `ValidationMetricPayload` (list or single run with nested `metrics[]`).
- Read from persisted `validation_runs` + `metric_results` (via `v_validation_summary` or repository).

#### Structured logging (verify/enhance)
- Confirm request ID middleware propagates to logs and error responses.
- Ensure audit-correlated actions log `request_id` where applicable.
- Data-quality evaluation counts should be measurable from persisted `quality_assessments` or validation `details` JSON.

### Step 4) Safety and security verification

Produce machine-checkable evidence (scripts + test suite + summary artifact).

1. **Secrets and real-data scan**
   - Scan codebase and committed seeds for API keys, passwords, PINs, OTP patterns, real phone numbers, and non-synthetic account identifiers.
   - Fail CI if matches found outside `.env.example` placeholders.

2. **Unsafe action endpoint scan**
   - Assert no routes named or behaving as transfer, convert, settle, refill, recover, reverse, block, freeze, accuse, or fraud-decision.
   - Reuse or extend existing route inventory tests.

3. **Prohibited language scan**
   - Scan user-visible strings (alert templates, explanation renders, frontend labels) for definitive fraud language ("fraud", "guilty", "criminal", "scammer", etc.).
   - Allowlist only in negative-test fixtures if needed.

4. **Authorization and immutability regression**
   - Re-run provider A/B denial tests (`tests/phase5/test_authorization.py`, `tests/phase6/test_provider_isolation_*`).
   - Verify append-only ledger/evidence/audit protections still hold.
   - Verify alert analytical evidence remains immutable after case workflow mutations.

5. **Responsible-design checks**
   - Bad data (Scenario C faults) must not produce misleading high-confidence projections or alertable anomaly candidates.
   - Suppressed evaluations remain measurable but non-alertable.

Write results to `docs/evidence/` (or `backend/reports/`):
- `validation-summary.json` — latest completed run with all metrics.
- `performance-reliability.json` — latency and completeness measurements.
- `safety-security-scan.json` — scan pass/fail with counts and any waivers.
