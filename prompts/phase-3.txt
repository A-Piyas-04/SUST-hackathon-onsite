You are a senior full-stack engineer. Implement **Phase 3 (Synthetic Ecosystem, Ingestion and Ledger)** for this repository as defined in `docs/16-hour-hackathon-phase-distribution.md`, while strictly following `docs/Problem_Statement.md`.

## Mission

Build a deterministic synthetic ecosystem and safe ingestion-to-ledger flow that produces trustworthy, replayable, provider-separated operational data for later quality, forecasting, anomaly, and workflow phases.

## Hard Guardrails (Never Violate)

1. Decision-support only; no autonomous financial or punitive action.
2. Never implement transfer, settlement, refill, recovery, reversal, freeze, block, accusation, or fraud verdict endpoints.
3. Shared physical cash and provider e-money (bKash/Nagad/Rocket) must remain separate at schema, service, and API layers.
4. Use synthetic/mock/demo-safe data only.
5. Preserve provider identity through every transformation.
6. Append-only protections for ledger/evidence tables must be respected.
7. Conflicting snapshots are preserved and visible (not silently overwritten).

## Scope Boundary (Phase 3 Only)

Implement only:
- synthetic scenarios + deterministic run/fault controls
- ingestion and normalization
- append-only ledger persistence and separated reads
- Phase 3 API group + tests

Do **not** implement full Phase 4/5 engines and workflows (full liquidity forecast engine, anomaly scoring engine, candidate-to-alert lifecycle, full case orchestration). You may keep contract-compatible stubs where needed.

2. Verify Phase 2 app foundation exists (`/health`, config, DB/session infra, base contracts).

## Implementation Plan (execute in this order)

### Step 1) Synthetic scenario architecture

Create a deterministic scenario module with:
- scenario catalog (`normal`, `A`, `B`, `C`, `D-ready`)
- parameterized seed + run configuration
- synthetic entity templates (area/outlet/provider account)
- deterministic clock strategy (fixed base timestamp + offsets)

Design constraints:
- Same input seed/config => same semantic dataset and run outcomes.
- Generated data must include one outlet, one shared cash reserve track, and three provider account tracks.
- Keep generation pure and testable (no hidden randomness without seeded source).

### Step 2) Run lifecycle and fault injection

Implement run orchestration:
- create run
- get run status
- reset run deterministically
- add fault
- toggle fault

Required fault types:
1. delayed feed
2. missing feed or missing field
3. malformed payload
4. conflicting snapshot

Each fault must include:
- ID, run ID, fault type, active flag
- deterministic parameters (window, target provider, rate, etc.)
- created/updated timestamps and actor metadata if available

### Step 3) Ingestion + normalization pipeline

Implement ingestion service with:
1. provider-specific mock payload adapters
2. normalized internal event model
3. strict validation + rejection path
4. ingest batch metadata persistence
5. event-level status recording

Normalization rules:
- unify structure but keep original provider identity and source references
- preserve source timestamps and ingestion timestamps separately
- attach trace IDs for replay/audit

Rejection rules:
- malformed/invalid events are persisted as rejected metadata but must **not** mutate ledger balances/transactions
- include explicit `rejection_reason` taxonomy

### Step 4) Ledger persistence and read repositories

Persist append-only:
- transactions
- shared cash snapshots
- provider balance snapshots

Implement repository/query layer for:
- latest balances per reserve type
- time-series history
- outlet dashboard composition
- conflict candidate visibility

Non-negotiables:
- enforce provider/account/outlet consistency
- never compute or expose blended "total monetary balance"
- preserve conflicting snapshots as coexisting records where schema supports

### Step 5) Phase 3 API surface

Implement and wire routes:
- `GET /api/v1/providers`
- `GET /api/v1/areas`
- `GET /api/v1/outlets`
- `GET /api/v1/outlets/{outletId}`
- `GET /api/v1/outlets/{outletId}/dashboard`
- `GET /api/v1/outlets/{outletId}/transactions`
- `GET /api/v1/outlets/{outletId}/balances/history`
- `GET /api/v1/simulations/scenarios`
- `POST /api/v1/simulations/runs`
- `GET /api/v1/simulations/runs/{runId}`
- `POST /api/v1/simulations/runs/{runId}/reset`
- `POST /api/v1/simulations/runs/{runId}/faults`
- `PATCH /api/v1/simulations/runs/{runId}/faults/{faultId}`
- `POST /api/v1/ingestion/batches`
- `GET /api/v1/outlets/{outletId}/data-quality`
- `GET /api/v1/outlets/{outletId}/data-quality/history`

Phase-3 data-quality endpoints:
- return foundation or interim outputs clearly marked as pre-Phase-4 estimates
- keep response contracts forward-compatible with Phase 4 quality model
