You are a senior full-stack engineer. Implement **Phase 6 (Integrated API and Thin UI)** for this repository as defined in `docs/16-hour-hackathon-phase-distribution.md`, while strictly following `docs/Problem_Statement.md`.

## Mission

Compose Phases 3–5 into one runnable MVP release candidate with a minimal, clear demonstration surface. A reviewer must be able to execute Scenarios A–D end to end through the integrated backend and thin frontend without direct database edits.

## Hard Guardrails (Never Violate)

1. Decision-support only. No real financial operations or punitive automation.
2. Never implement transfer/refill/recovery/reversal/freeze/block/accusation/fraud verdict actions.
3. Shared physical cash and provider e-money (bKash, Nagad, Rocket) remain separate in every API response and UI card — never blended or summed into one total.
4. Provider boundaries are strict: Provider A users cannot read or mutate Provider B confidential records.
5. Alert analytical evidence is immutable; case workflow state is mutable and auditable.
6. Use safe wording everywhere: "unusual", "requires review", uncertainty-aware language — never definitive fraud claims.
7. For confidential resources, forbidden and missing return the same safe-not-found style response.
8. Use synthetic/demo-safe identities and data only.
9. Suppressed/degraded analytical results must not present as high-confidence actionable alerts in the UI.

## Scope Boundary (Phase 6 Only)

Integrate existing backend modules, complete the thin demo frontend, and add end-to-end regression coverage.

Do **not** implement:
- Phase 7 validation/metrics evidence (`/metrics`, `/api/v1/validation/results`, held-out evaluation reports)
- Phase 8 documentation deliverables (README expansion, architecture diagram, presentation slides)
- Stretch endpoints after the MVP gate (`what-if-runs`, `relationships`, `nearby-support-options`, `support-requests`)
- Visual polish beyond what is needed for a reliable demo

## Implementation Plan (execute in this order)

### Step 1) Backend composition and release readiness

1. Register all Phase 3–5 routers through one application composition path (`app/api/router.py` or equivalent).
2. Verify every confidential endpoint uses the central `current_user` auth dependency — no bypasses.
3. Remove or clearly mark any stale stubs that claim unimplemented behavior works.
4. Verify startup/shutdown hooks, CORS for the frontend origin, structured logging with request IDs, and safe global error handling.
5. Confirm documented commands work end to end:
   - migrations (`migrate`, `seed`, `reset`)
   - server start
   - OpenAPI generation (`openapi` → `docs/openapi/openapi.v1.json`)
   - backend test suite (`test`)
6. Freeze the OpenAPI contract for this release candidate and ensure it matches registered routes.

### Step 2) Thin responsive frontend shell

Build on the existing Next.js scaffold in `frontend/`. Extend `src/lib/api.ts` with typed fetch helpers for all MVP endpoints. Use environment variable `NEXT_PUBLIC_API_BASE_URL` (default `http://localhost:8000`).

Required UI surfaces (thin but complete — clarity over polish):

1. **Demo login / role switch**
   - login as representative roles (agent, field/area ops, provider ops, risk analyst, management)
   - show current user, role, and scope context
   - persist auth token/session for subsequent requests

2. **Outlet dashboard**
   - shared-cash card and three separate provider balance cards (bKash, Nagad, Rocket)
   - data-health indicator per feed (fresh/stale/missing/conflicting)
   - confidence states on projections and anomaly results
   - never display a blended monetary total

3. **Liquidity intelligence panel**
   - shortage timing estimate per reserve
   - confidence level/score on every projection
   - contributing signals breakdown
   - non-actionable/degraded states rendered safely (not as confident shortage)

4. **Anomaly evidence panel**
   - unusual-pattern summary with structured evidence (counts, amount cluster, parties, window)
   - plausible benign context displayed prominently
   - suppressed evaluations visible for measurement but clearly marked non-alertable

5. **Scenario and fault controls**
   - scenario list and run start/status/reset
   - fault create/toggle for live Scenario C demonstration
   - trigger analytics runs (liquidity + anomaly) after simulation run

6. **Alert explanations**
   - alert list/detail
   - locale toggle between English and Bangla/Banglish
   - situation, evidence, uncertainty, next step, and benign context sections

7. **Case workflow controls**
   - open case from alert
   - assignment display, acknowledge, escalate, add note, review, resolve
   - current owner, recommended next step, and status visible at all times

8. **Notifications and audit**
   - in-app notification list with read/unread state
   - case timeline and audit event display with clear ordering

### Step 3) UI state handling

Every data-fetching surface must implement:

1. **Loading** — skeleton or spinner while fetching
2. **Empty** — explicit empty state when no data exists yet (e.g., no alerts, no notifications)
3. **Error** — safe error message with retry where appropriate
4. **Forbidden** — same safe presentation as not-found for confidential resources (no "access denied" leakage that reveals existence of cross-provider records)

Use polling (3–5 second interval) for dashboard refresh during demo; WebSockets are not required.

### Step 4) End-to-end scenario wiring

Wire deterministic demo paths that mirror existing backend tests:

#### Scenario A — Hidden provider shortage
1. Start `scenario_a` simulation run with deterministic seed.
2. Run liquidity analytics.
3. UI shows shared cash depleting with shortage timing and confidence.
4. Provider e-money balances remain separate; target provider does not falsely show e-money shortage.
5. Alertable liquidity candidate is visible.

#### Scenario B — Unusual activity with benign context
1. Start `scenario_b` simulation run.
2. Run anomaly analytics.
3. UI shows near-identical-amount pattern with structured evidence.
4. Benign context explanation is visible.
5. Publish alert and show immutable evidence in alert detail.

#### Scenario C — Degraded data lowers confidence and suppresses alerts
1. Start `scenario_c` simulation run (with conflicting/degraded feed).
2. Run quality + liquidity + anomaly analytics.
3. UI shows reduced confidence and conflicting data-health state.
4. Suppressed anomaly evaluation is visible but not presented as a high-confidence alert.
5. Fault controls allow live demonstration of degraded feed behavior.

#### Scenario D — Coordinated response and closure
1. Publish alert from Scenario B analytics (or equivalent).
2. Open routed case with owner and next step.
3. Walk full lifecycle through UI: acknowledge → note → escalate → review → resolve.
4. Timeline and audit history are complete and queryable in UI.
5. Alert evidence remains unchanged after case mutations.
6. No direct database edits at any step.

### Step 5) Integration verification

Re-run and extend tests to cover:

1. **Provider isolation** — cross-provider list/detail/mutation denied in UI-driven API calls
2. **Alert immutability** — evidence unchanged after workflow mutations
3. **Idempotency** — duplicate mutations fail safely without duplicated side effects
4. **Concurrency** — stale version conflicts handled gracefully
5. **Safe language** — scan user-visible strings for definitive fraud language
6. **No blended totals** — dashboard and projection views never sum reserves

### Step 6) One-command runnable release

Provide documented commands to start the full stack:

1. Backend: migrate + seed + server (existing `backend/Makefile` targets or equivalent)
2. Frontend: `npm run dev` (or `npm run build && npm start` for production-like demo)
3. Optional root-level convenience script or documented two-terminal workflow

Add an end-to-end regression suite (Playwright, Cypress, or API+UI integration tests) that exercises Scenarios A–D deterministically.

Capture backup API response snapshots or screenshots for critical demo steps (for Phase 8 presentation fallback).
