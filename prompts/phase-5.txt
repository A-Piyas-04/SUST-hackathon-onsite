You are a senior full-stack engineer. Implement **Phase 5 (Alerts, Cases and Security)** for this repository as defined in `docs/16-hour-hackathon-phase-distribution.md`, while strictly following `docs/Problem_Statement.md`.

## Mission

Turn validated analytical candidates into immutable, localized alerts and provider-scoped human coordination workflows with enforceable authorization, legal transitions, notifications, and complete auditability.

## Hard Guardrails (Never Violate)

1. Decision-support only. No real financial operations or punitive automation.
2. Never implement transfer/refill/recovery/reversal/freeze/block/accusation/fraud verdict actions.
3. Keep provider boundaries strict: Provider A users cannot read/mutate Provider B confidential records.
4. Shared cash and provider reserves remain conceptually separate in every response.
5. Alert evidence is immutable; case workflow state is mutable and auditable.
6. Use safe wording: "unusual", "requires review", uncertainty-aware language.
7. For confidential resources, forbidden and missing should return same safe-not-found style response.
8. Use synthetic/demo-safe identities and data only.

## Scope Boundary (Phase 5 Only)

Implement auth + alert + case + notification + audit API/domain behavior.  
Do **not** implement full UI experience (Phase 6). Thin test controls are allowed.

## Prerequisite Validation (do first)

1. Confirm Phase 4 persisted results + valid `AlertCandidate` flow exist.
2. Confirm RLS expectations and scope models from schema are available.
3. If candidate lineage or access-scope schema is missing, stop and report blockers.

### Step 1) Authentication and principal context

1. Demo identity seeds for representative roles:

   - management

### Step 2) Authorization and provider boundaries

Implement reusable authorization guards:
1. provider scope checks
2. outlet/area hierarchy checks
3. optional role capability checks

Behavior:
- deny cross-provider confidential access
- missing and forbidden confidential IDs return same safe-not-found response body/status policy
- align with DB RLS assumptions (application and DB boundaries reinforce each other)

### Step 3) Candidate-to-alert publication

Implement alert publication flow:
1. validate source candidate integrity, scope, and alertability
2. deduplicate active alerts
3. persist typed source links (no evidence recomputation drift)
4. publish immutable analytical alert payload
5. preserve confidence/uncertainty and benign context references

Immutability rule:
- once published, analytical evidence fields cannot be edited through workflow endpoints

### Step 4) Localized explanation rendering

Implement explanation templates/renders for:
1. English
2. Bangla/Banglish (at least one practical demo template)

Each rendered explanation should include:
- situation
- supporting evidence
- uncertainty/confidence
- safe next step
- plausible benign context (where relevant)

Persist explanation render snapshots linked to alert/version context.

### Step 5) Routing and case lifecycle

Routing precedence must be:
1. provider + area
2. provider fallback
3. area fallback
4. default fallback

Implement case domain:
- open case from alert
- assignment/reassignment
- acknowledge
- escalate
- add note
- review
- resolve

State rules:
1. explicit legal transition matrix
2. resolution preconditions enforced
3. optimistic concurrency/version checks
4. idempotency keys or equivalent duplicate mutation defense

### Step 6) Notifications and audit trail

Implement:
1. in-app notification list/read behavior
2. atomic audit-event write with each workflow mutation
3. case timeline composition from status/actions/audit
4. auditable actor/action/when/why/context fields

Audit requirements:
- immutable historical events
- clear ordering semantics
- no missing event for high-impact state changes
