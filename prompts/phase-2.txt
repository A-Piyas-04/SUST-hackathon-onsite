You are a senior full-stack engineer. Implement **Phase 2 (Application Foundation and Contracts)** for this repository as defined in `docs/16-hour-hackathon-phase-distribution.md`, while strictly following `docs/Problem_Statement.md`.

## Mission

Build a runnable, production-style **modular monolith foundation** (backend-first, thin frontend shell) on top of the already-completed Phase 1 schema, and freeze v1 contracts so later phases can be developed safely and fast.

## Non-Negotiable Domain Guardrails

1. Treat this as a **decision-support** system only.
2. Never implement any real financial action (no transfer/settlement/refill/recovery/reversal/freeze/block/accusation).
3. Preserve strict separation between:
   - shared physical cash reserve
   - provider-specific e-money reserves (bKash, Nagad, Rocket)
4. Use only synthetic/mock/demo-safe data.
5. Use safe language: anomaly means “unusual” / “requires review”, never final fraud verdict.
6. Design auth boundary now; do not allow confidential routes to bypass auth dependency.

## Scope: Implement Phase 2 Only

Do not implement Phase 3+ business features yet (no full ingestion engine, no forecasting logic, no anomaly engine, no case workflow UI). You may add stubs/placeholders only where required for contracts/OpenAPI.

## Required Outcomes

### A) Backend foundation

Create/organize backend into clear modules (example):
- `app/main.py` (or equivalent app bootstrap)
- `app/core/` (config, logging, errors, request-id, security interfaces)
- `app/db/` (engine/session/transaction/migration hooks)
- `app/api/` (versioned routers)
- `app/contracts/` (Pydantic schemas / DTOs)
- `app/services/` (minimal orchestration stubs)
- `tests/`

Use FastAPI + Pydantic + SQLAlchemy/Alembic (or equivalent stack already present in repo if different). Keep structure clean and phase-ready.

### B) Configuration and environment

1. Add robust env loading with validation.
2. Add `.env.example` with placeholders only (no secrets, no real credentials).
3. Include required vars for DB URL, app env, log level, CORS origins, and optional feature flags.
4. Fail fast with clear error messages if mandatory env vars are missing.

### C) Database/session/transaction infrastructure

1. Async DB connection support.
2. Session lifecycle dependency for request-scoped DB usage.
3. Explicit transaction helper pattern with rollback safety.
4. Readiness check that validates DB connectivity and critical schema availability.

### D) App lifecycle, CORS, request ID, logging, global error handling

1. Implement app factory pattern.
2. Add startup/shutdown hooks.
3. Configure CORS for thin frontend integration.
4. Add request correlation ID middleware.
5. Structured logs for request start/end/error with request ID.
6. Global exception handlers that:
   - return safe error payloads
   - avoid leaking internals
   - preserve traceability in logs

### E) Authentication dependency interface (foundation only)

1. Define `get_current_user`-style dependency interface and user context model.
2. Add pluggable auth provider abstraction (mock/demo-ready).
3. Ensure confidential endpoints are wired to auth dependency even if logic is stubbed.
4. Do not add auth bypasses that would violate future provider boundaries.

### F) Versioned contracts (freeze v1 seam contracts)

Create strongly typed v1 contracts for:
1. normalized transaction input
2. normalized balance snapshot input
3. quality assessment
4. liquidity projection
5. anomaly flag + evidence
6. `ResultEnvelope`
7. `AlertCandidate`
8. validation metric payload
9. dashboard response
10. alert response
11. case response

Requirements:
- strict field typing and validation constraints
- enums where appropriate
- timezone-safe timestamps
- decimal-safe money fields
- confidence and evidence structures explicit
- `AlertCandidate` must be derivable from `ResultEnvelope`
- keep case workflow mutable fields out of candidate model

### G) Fixtures + contract tests

1. Add positive and negative fixtures for each major contract.
2. Implement automated tests validating:
   - schema validation pass/fail behavior
   - required-field and enum constraints
   - decimal/timestamp parsing robustness
   - `ResultEnvelope -> AlertCandidate` adapter correctness
3. Tests must be runnable from one documented command.

### H) API and OpenAPI baseline

1. Register at least:
   - `GET /health` (Phase 2 required)
2. Add minimal stub routers (versioned under `/api/v1`) only as contract placeholders where useful.
3. Generate OpenAPI spec from actual registered routes.
4. Do not claim unimplemented business features are complete; mark stubs clearly.

### I) Commands and developer workflow

Document one consistent command set (e.g., Makefile / npm scripts / task runner / README):
- run server
- run migrations
- run seed
- run tests
- generate OpenAPI

Commands must work in a fresh setup against completed Phase 1 schema.

### J) Frontend shell only

Create minimal frontend scaffold (no feature implementation), enough to:
- install/run
- connect to backend base URL config
- be ready for Phase 6 UI work

## Acceptance Criteria (must all pass)

1. App boots cleanly and `GET /health` confirms DB readiness.
2. Migration/seed/test/openapi commands run from documented workflow.
3. Contract tests pass for positive and negative fixtures.
4. Valid `ResultEnvelope` can be transformed into valid `AlertCandidate` fixture.
5. No confidential route bypasses auth dependency.
6. OpenAPI baseline is generated and committed.
7. No code violates challenge safety boundaries in `docs/Problem_Statement.md`.

1. Keep modules cohesive and future-phase extensible.
2. Prefer explicit interfaces over hidden coupling.
3. Add concise docstrings/comments for non-obvious logic only.
4. Use consistent naming and error taxonomy.
5. Keep CI-friendly deterministic tests.
6. If repository already has conventions, follow them.
