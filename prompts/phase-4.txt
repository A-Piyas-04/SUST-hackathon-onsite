You are a senior full-stack engineer. Implement **Phase 4 (Data Quality and Intelligence)** for this repository as defined in `docs/16-hour-hackathon-phase-distribution.md`, while strictly following `docs/Problem_Statement.md`.

## Mission

Transform Phase 3 ledger outputs into explainable, confidence-aware analytical intelligence by implementing:
1. Data Quality & Confidence Engine
2. Liquidity Forecasting Engine
3. Anomaly Detection Engine
4. Persistence + evidence APIs + `ResultEnvelope -> AlertCandidate` seam

## Hard Guardrails (Never Violate)

1. Decision-support only; no autonomous punitive/financial action.
2. Shared cash and provider reserves are modeled and analyzed separately.
3. Missing/stale/conflicting data must reduce confidence and may suppress alertability.
4. Anomaly is "unusual/requires review", never final fraud proof.
5. Suppressed anomaly evaluations may be stored for measurement, but cannot create high-confidence anomaly candidates.
6. Analytical outputs must be reproducible and evidence-backed.
7. Use synthetic/mock/demo-safe data only.

## Scope Boundary (Phase 4 Only)

Implement analytics engines, persistence, API access, and candidate adapter seam.  
Do **not** implement full Phase 5 auth/routing/case workflow/notification lifecycle.

## Prerequisite Validation (do first)

1. Confirm Phase 3 normalized inputs and ledger reads exist and are stable.
2. Confirm Phase 2 contracts (`ResultEnvelope`, `AlertCandidate`) exist or can be safely finalized.
3. If input reliability or schema links are missing, stop and report blockers precisely.

## Implementation Plan (execute in this order)

### Step 1) Data Quality & Confidence Engine

Implement classification over analytical inputs:
- `fresh`
- `stale`
- `missing`
- `conflicting`

Required behavior:
1. Define explicit precedence when multiple issues coexist.
2. Emit issue evidence with machine-readable reason codes.
3. Include sample count, last source timestamp, confidence modifier, and trusted-value metadata.
4. Preserve conflicting candidates explicitly.
5. Handle malformed/insufficient input as safe non-actionable outcomes.

Outputs must be deterministic, explainable, and traceable to source windows.

### Step 2) Liquidity Forecasting Engine

Implement transparent forecasting independently for:
1. shared physical cash reserve
2. each provider reserve per outlet

Algorithm expectations:
- recent-window depletion/burn-rate logic (explicit and inspectable)
- no-shortage outcome when depletion <= 0
- non-actionable outcome when sample sufficiency fails
- confidence derived from quality modifiers + sample adequacy
- uncertainty bounds widening under degraded quality
- contributing signals returned in response payload

Strictly no blended reserve forecast.

### Step 3) Anomaly Detection Engine

Implement at least one concrete rule:
- near-identical amount repetition within same provider/outlet/time window

Required outputs:
1. evidence references to source transactions/events
2. count metrics and amount-cluster summary
3. distinct party/account count
4. detection window
5. confidence + reason
6. plausible benign context explanation

Suppression behavior:
- degraded data can trigger suppression status
- suppressed evaluations are persisted with suppression reason
- suppressed evaluations cannot become alertable anomaly candidates

### Step 4) Persistence integration

Persist all analytical artifacts into schema structures:
- analytics runs
- quality assessments/issues
- liquidity projections/signals
- anomaly flags/evidence and transaction links

Requirements:
1. run-level lineage and reproducibility metadata (seed, config, algorithm version where available)
2. immutable evidence references
3. decimal/timestamp precision preserved end-to-end

### Step 5) API implementation (Phase 4 delivery map)

Implement:
- `GET /api/v1/outlets/{outletId}/liquidity-projections`
- `POST /api/v1/internal/analytics/liquidity/run`
- `GET /api/v1/outlets/{outletId}/anomaly-flags`
- `GET /api/v1/anomaly-flags/{flagId}`
- `POST /api/v1/internal/analytics/anomalies/run`

Also keep Phase 3 quality endpoints forward-compatible with quality engine outputs.

### Step 6) Result contracts and adapter seam

Implement strict adapter:
- `ResultEnvelope -> AlertCandidate`

Rules:
1. preserve analytical source identity and evidence links.
2. include confidence, uncertainty, and alertability metadata.
3. do not include mutable case-workflow fields in candidate.
4. suppressed/non-actionable results must not produce high-confidence anomaly candidates.

## Contract and Data Rules

1. Money uses decimal-safe types.
2. Timestamps are timezone-aware ISO 8601.
3. Confidence is explicit numeric + optional categorical level.
4. Evidence payloads are structured (not free-form-only strings).
5. Engine versions/rule IDs should be traceable in outputs where possible.
